intellisource-ai — reading the source

From main.py to report.html, one stage at a time

This tool is one strict, linear pipeline: clone a repo, parse it for free (structure, complexity, security findings, dependency graph, git churn), ask an LLM what the code means, cache that, and write three files. Below is the exact call chain, file by file, plus every terminal command you need to set it up, run it, and test it.

Setup The Pipeline How It's Wired Running It Running Tests Cheat Sheet

1. Setup — do this once

Three commands: install the package in editable mode, create your local config file, then fill in two values.

PowerShell
PS> pip install -e ".[dev]"
# installs intellisource-ai itself, plus pytest/ruff/mypy
 
PS> copy .env.example .env
# then open .env and fill in:
# ANTHROPIC_API_KEY=sk-ant-...
# REPO_URL=https://github.com/<owner>/<repo>

That's it — no database, no server to start. .env is read once, at the top of every run, by config.py.

2. The pipeline, stage by stage

run_pipeline() in pipeline.py calls each of these in order — no stage ever calls backward to a previous one. Each has exactly one file, and does exactly one job.

0
main.py

Entry point

Parses CLI flags, resolves settings, turns on logging, then hands off to the pipeline. Deliberately has no logic of its own beyond that hand-off — if something breaks here, the fix lives elsewhere.

calls resolve_settings(args), then run_pipeline(settings)
1
config.py

Resolve settings

Merges CLI flag → environment variable → .env, in that order of priority. Fails immediately, before any network call, if REPO_URL or ANTHROPIC_API_KEY is missing from all three sources.

resolve_settings() returns a frozen Settings object every later stage reads from
2
repo_fetcher.py

Clone the target repo — or skip this entirely

Shells out to the system git for a clone, depth controlled by --git-history-depth (default 200 commits — enough for real churn history, not a full clone), into .cache/<repo-name>/repo. Reuses that folder on the next run unless --refresh-repo is passed. If --local-path <dir> is given instead (e.g. a GitHub Action step running after actions/checkout), this whole stage is skipped and that directory is used as-is.

fetch_repository() returns the local path to the checked-out code
3
java_parser.py

Parse every .java file — free

Walks src/main/java, runs each file through the javalang grammar parser, and extracts class names, method signatures, annotations, and REST routes as plain structured data. No LLM call happens here. A file that fails to parse is logged and skipped, never aborts the run.

parse_source_tree() returns a ParseResult — a list of ParsedClass
4
complexity.py

Score every method — also free

For each method's source lines: counts branching keywords (if/for/while/case/catch/&&/||) to get a rough cyclomatic estimate, and flags a method as high-complexity if that estimate or its line count crosses a threshold.

called once per method from pipeline._compute_all_complexity()
5
security_scanner.py

Flag hardcoded secrets, SQL concatenation, empty catches — still free

Scans each class's whole body text (fields included, not just method bodies — a hardcoded secret is usually a field) for three regex-based patterns: credential-like field values, SQL strings built via + concatenation, and empty/comment-only catch blocks. Same "documented heuristic" posture as the complexity estimate above, not a certified security scanner.

scan_class() returns a list of SecurityFinding, called from pipeline._scan_all_security()
6
dependency_graph.py

Build the internal dependency graph — also free

Reads the import statements java_parser.py already captured, keeps only the ones whose simple class name matches another class parsed this run (external JDK/framework/library imports are dropped), and builds directed from_class → to_class edges. Later ranked by fan-in to surface the most relied-upon classes in the report.

build_dependency_graph() → edges; depends_on_by_class() groups them per class
7
churn.py

One git log pass for commit frequency — still free

Runs a single git log --name-only against the checked-out repo and counts commits per file. Re-anchors paths to repo_root rather than the git top level, so a --local-path pointed at a subdirectory of a larger repo still lines up with every other stage's file_path. Never raises — a shallow clone, a non-git directory, or a missing git binary all just degrade to "no churn data" for that run.

compute_churn() returns {file_path: ChurnMetrics}; combined with complexity in pipeline._compute_hotspots()
8
chunker.py

Group classes into LLM-sized batches

Renders each class into condensed, signatures-only text (never raw source), groups classes by directory, then packs them into batches capped by both --batch-size and a real token count — measured against the actual Anthropic tokenizer, not guessed from character length.

build_batches() returns a list of ClassBatch, each with one prompt string
9
cache.py + llm_client.py

Ask the LLM what the code means

Before any call, each class's rendered text is hashed and checked against cache.py's on-disk cache — an unchanged class costs zero tokens on a re-run. Everything else goes to llm_client.py, which sends the batch to Claude via LangChain and gets back a schema-guaranteed ClassBatchAnalysis (one description + notable aspects per class, one sentence per method). This is the only stage that costs money.

LLMCache.get() first, LLMClient.analyze_batch() on a miss, LLMCache.set() to store the result
10
pipeline.py

Assemble the final record

Merges the parser's structure, the complexity scores, the LLM's descriptions, and the three deterministic signals from stages 5–7 (security findings, dependency edges, churn) back together into one ClassAnalysis per class. A class whose LLM call failed gets an explicit "Description unavailable." placeholder rather than being silently dropped. One more LLM call — generate_overview() — produces the single project-wide summary from the README, build file, and every class's one-line purpose. Churn x complexity is then ranked into a hotspots list of the riskiest classes.

_assemble_classes() + _generate_overview() + _compute_hotspots() → one ProjectAnalysis
11
report_generator.py

Write the three output files

ProjectAnalysis is serialized straight to output/analysis.json; its Pydantic schema is dumped to output/analysis.schema.json; and the same object is rendered through a Jinja2 template into output/report.html. All three come from one object, so they can never disagree with each other.

_write_outputs() — the last thing run_pipeline() does

3. How the pieces are wired together

A few connective details that aren't obvious from reading one file at a time.

intellisource_ai.schemas
Every arrow in the diagram above passes a Pydantic model, never a raw dict — ParsedClass → ComplexityMetrics → ClassDescription → ClassAnalysis. If you want to know the exact shape of data moving between two stages, this file is the contract.
intellisource_ai.exceptions
Each stage raises its own named exception (RepoFetchError, JavaParseError, LLMExtractionError) instead of a bare Exception. pipeline.py catches the ones it knows how to recover from — one bad file or one failed batch — and lets configuration errors propagate all the way up to main.py, which prints a clean message instead of a traceback.
PROMPT_VERSION (cache.py)
The cache key is a hash of the prompt text plus this version string. Change the extraction prompt in llm_client.py, bump this constant, and every cached result is invalidated at once — nothing is silently mixed across prompt versions.
output/ vs .cache/
.cache/ holds the cloned repo and the LLM response cache — disposable, regenerated on demand. output/ holds the three deliverables described in stage 11 above — what you actually came here for.
hotspots = churn x complexity
pipeline._compute_hotspots() scores every class as churn.commit_count * (1 + high-complexity method count) and keeps the top 8. A class with no churn data (history unavailable) scores 0 and is excluded rather than favored — "risky" here always means both signals agreeing, not either alone.
action.yml — the same pipeline, as a GitHub Action
The repo root's action.yml is a composite action wrapping this exact CLI: it runs main.py --local-path $GITHUB_WORKSPACE (stage 2 skipped, since actions/checkout already did the equivalent), caches .cache/ across runs via actions/cache, and writes a job-summary table from analysis.json's metadata. See the README's "Use as a GitHub Action" section for the copy-paste workflow snippet.

4. Running it

One required flag (or set it once in .env and skip it forever after).

PowerShell
# full run against a repo
PS> python main.py --repo-url https://github.com/<owner>/<repo>
 
# cheap smoke test — only analyze the first 15 files
PS> python main.py --repo-url https://github.com/<owner>/<repo> --max-files 15
 
# force a fresh clone and bypass the LLM cache
PS> python main.py --repo-url https://github.com/<owner>/<repo> --refresh-repo --refresh-cache
 
# analyze a directory already on disk -- no clone at all
PS> python main.py --local-path C:\path\to\checked-out\repo
 
# clone more history so churn/hotspots reflect real activity
PS> python main.py --repo-url https://github.com/<owner>/<repo> --git-history-depth 500
 
# see every flag
PS> python main.py --help

Watch the log lines as it runs — each stage above logs its own progress ("Discovered N Java file(s)", "N class(es) served from cache", "Grouped N classes into N LLM batch(es)"), so you can literally see this walkthrough happen in real time.

5. Running the tests

The whole suite runs offline — no API key needed, nothing hits the network. Every LLM and tokenizer call is faked in tests/conftest.py.

PowerShell
# everything
PS> pytest
 
# one file, verbose
PS> pytest tests/test_java_parser.py -v
 
# lint
PS> ruff check .
 
# strict type-check
PS> mypy src
A good first step: open tests/test_java_parser.py next to src/intellisource_ai/java_parser.py. Each test feeds a small embedded Java string in and asserts on the exact ParsedClass that comes out — the clearest, smallest example of "input in, structured data out" in the whole codebase.
