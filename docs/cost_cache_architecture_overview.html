Architecture Walkthrough

How IntelliSource AI turns a repo into a report

Thirteen stages, entry point to output. Two colors do the real work here: teal marks a step that costs nothing — pure parsing and bookkeeping — and amber marks the only two steps that actually call the LLM and cost tokens.

1

Entry point main.py

Free

python main.py --repo-url <url> calls main(). It's deliberately thin: parse CLI args → configure logging → resolve settings → call run_pipeline(). No business logic lives here by design.

Technicallymain.py calls build_arg_parser() and parser.parse_args() to read your CLI flags.
2

Config resolution config.py: resolve_settings()

Free

Merges CLI flags, environment variables, and .env, in that precedence order. Fails fast here — before any network call — if repo_url or ANTHROPIC_API_KEY is missing, raising ConfigurationError. Returns an immutable Settings dataclass.

Technicallyconfig.py: resolve_settings() merges CLI args → env vars → .env, and raises ConfigurationError if repo_url or ANTHROPIC_API_KEY is missing.
3

Repo fetch repo_fetcher.py: fetch_repository()

Free

Shells out to git clone --depth <N> --branch <ref>, where N is --git-history-depth (default 200 — enough for real churn history, not a full clone). Reuses an existing clone under .cache/<repo-name>/repo unless --refresh-repo is passed. Skipped entirely when --local-path is given instead.

Technicallyrepo_fetcher.py: fetch_repository() shells out to git clone --depth <git_history_depth> --branch <ref> into .cache/<repo-name>/repo, or is bypassed when settings.local_path is set.
4

Java parsing java_parser.py: parse_source_tree()

Free

Walks the repo's .java files (excluding src/test/java unless --include-tests), uses javalang to build an AST per file. Recurses into nested static classes. Extracts package, class type (heuristic: Spring stereotype annotations first, directory keywords as fallback), method signatures, annotations, and REST routes from @RequestMapping/@GetMapping etc. A file that fails to parse (e.g. unsupported syntax) is logged into parse_errors and skipped — the run continues.

Technicallyjava_parser.py: parse_source_tree() runs each file through the javalang AST parser, extracting classes, methods, annotations, and REST routes; failures are caught and logged to parse_errors.
5

Complexity computation complexity.py

Free

For every method, computes LOC and a heuristic cyclomatic complexity — 1 + count of if/for/while/case/catch/&&/|| on the method body, with strings/comments stripped first so keywords inside them don't get miscounted. Purely regex-based, zero LLM cost.

Technicallycomplexity.py, called from pipeline._compute_all_complexity(), computes LOC and a regex-based cyclomatic estimate (1 + count of if/for/while/case/catch/&&/||) per method.

Also free, added later: security_scanner.py flags hardcoded secrets, SQL built via concatenation, and empty catch blocks; dependency_graph.py builds an import-based internal class graph; churn.py adds one git log pass for commit frequency, feeding the hotspots ranking (churn × complexity) computed alongside step 12. All three run in this same free phase — zero LLM cost — and degrade to empty rather than failing the run when unavailable (shallow clone, non-git --local-path, etc).

6

Cache check cache.py: LLMCache.get()

Free

For every parsed class, pipeline._analyze_classes() renders its condensed prompt text and hashes it (SHA-256(PROMPT_VERSION + rendered_text)). If that exact hash is already in the on-disk cache, the class is served with zero LLM cost. Only cache misses go on to the next step.

Technicallypipeline._analyze_classes() hashes each class's rendered prompt text (cache.compute_cache_key) and looks it up via LLMCache.get(), an on-disk JSON keyed by SHA-256(PROMPT_VERSION + text).
Cache hitReuse the saved description — skip straight to step 12, zero cost.
Cache missEnter the AI path below (steps 7–11).
AI path · only for cache misses
7

Batching chunker.py: build_batches()

Prep

Cache-miss classes get grouped by source directory (so a batch shares feature/layer context), then capped two ways: a max class count (--batch-size, default 6) and a real token ceiling (default 4,000 input tokens) — measured via the Anthropic SDK's actual count_tokens endpoint, never a character-count guess. A single class too large on its own gets its methods split across sub-batches instead of truncated.

Technicallychunker.py: build_batches() groups cache-misses by source directory, capped by --batch-size and a real token ceiling.
8

Prompt rendering chunker.py: render_class_for_prompt()

Prep

Each class is rendered into condensed signature/annotation/Javadoc/complexity-flag text — never raw source code. This is the key cost control: the LLM only sees what it needs to judge meaning, not syntax a parser already extracted for free.

Technicallychunker.py: render_class_for_prompt() renders each class into condensed signature/annotation/Javadoc/complexity text — never raw source — and _count_tokens() verifies size via the Anthropic SDK's count_tokens endpoint.
9

LLM call llm_client.py: LLMClient.analyze_batch()

AI call

Wraps a single ChatAnthropic (LangChain) instance with .with_structured_output(ClassBatchAnalysis), so the API/SDK — not prompt wording — guarantees schema-conformant JSON back. Token usage from each call is recorded onto a shared UsageTracker.

Technicallyllm_client.py: LLMClient.analyze_batch() calls a LangChain ChatAnthropic chain with .with_structured_output(ClassBatchAnalysis), guaranteeing schema-conformant JSON back.
10

Merge results back pipeline.py: _process_batch() / _merge_fragments()

Merge

Matches each returned class_name back to its source ParsedClass by exact string match. If a class was split across sub-batches, its method fragments get merged into one ClassDescription. A whole-class result (not split) gets written back into the cache via cache.set().

Technicallypipeline._process_batch() matches returned class_name values to source ParsedClass objects; _merge_fragments() recombines any class split across sub-batches.
11

Project overview pipeline.py: _generate_overview()

AI call

One additional LLM call: feeds a truncated README excerpt, a truncated build-file excerpt, and every class's one-line purpose (already generated in step 9) to LLMClient.generate_overview(), producing the project-wide summary, tech stack, and architecture description.

Technicallypipeline._generate_overview() feeds a truncated README/build-file excerpt plus all per-class summaries into LLMClient.generate_overview().
↓ rejoins the main line at step 12, alongside every cache hit from step 6
12

Assembly pipeline.py: _assemble_classes()

Free

Merges three independently-produced pieces of data per class — structural data from step 4, complexity metrics from step 5, and LLM descriptions from step 10 — into the final ClassAnalysis list. Any class or method the LLM never described gets an explicit "Description unavailable." placeholder rather than being silently dropped.

Technicallypipeline._assemble_classes() merges ParsedClass (structure), ComplexityMetrics, and ClassDescription into the final ClassAnalysis list, defaulting missing descriptions to "Description unavailable.".
13

Output generation pipeline.py: _write_outputs(), report_generator.py

Free

Everything assembles into one Pydantic model, ProjectAnalysis, which is the single source of truth rendered three ways: analysis.json (the data), analysis.schema.json (its JSON Schema, generated from the same Pydantic model), and report.html (Jinja2, autoescaped, self-contained). Since JSON and HTML both come from the same in-memory object, they can never disagree. A final summary line logs files parsed, cache hits vs. LLM calls, tokens, and estimated cost.

Technicallypipeline._write_outputs() serializes the single ProjectAnalysis Pydantic model to analysis.json and analysis.schema.json (via model_json_schema()), and report_generator.write_report() renders it through an autoescaped Jinja2 template into report.html.

What lands in output/

analysis.jsonThe structured deliverable — every class, method, and the project overview.
analysis.schema.jsonThe formal JSON Schema, generated from the same Pydantic model.
report.htmlA self-contained, browsable report of the same data.

* Figures above are from a real full run against spring-rest-sakila: 187 of 188 discovered files parsed (1 skipped for unsupported syntax), 246 classes analyzed, 3 LLM calls made because 235 classes were already cached from a prior run.

The bottom line

How do you keep this cheap and scalable?

Lead with step 4 vs. step 9 — splitting free, deterministic static analysis (parsing, complexity) from paid LLM calls (semantic judgment only), plus the content-hash cache in step 6 that makes repeat runs cost nothing for unchanged code.

It reads the code for free using normal parsing, only pays to ask an AI the one thing parsing can't tell you — what the code means — and remembers past answers so it never pays twice for the same class.
